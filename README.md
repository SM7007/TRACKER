# Tracker — Learning Progress Tracker

A small CMS-style app for tracking what you're learning:

```
User
 └── Course            (e.g. "React")
      └── Topic         (e.g. "Hooks")
           └── Subtopic  (e.g. "useEffect")  ← you tick these off
```

Tick a subtopic → its topic auto-marks **complete** once every subtopic under
it is done → its course auto-marks **complete** once every topic is done.
Everything (titles, courses, topics, subtopics) is editable in place, and
data is scoped per logged-in user.

**Stack:** React + Vite + Tailwind (client) · Node.js + Express (server) ·
Prisma + PostgreSQL on Neon (database) · JWT auth.
**One project, one `package.json`, one `npm install`, one server in
production.**

```
learn-tracker/
├─ package.json        ← the only package.json
├─ vercel.json          ← Vercel config (rewrites /api/* to the function below)
├─ api/
│   └─ index.js         ← Vercel serverless entrypoint (wraps server/index.js)
├─ prisma/
│   └─ schema.prisma
├─ server/              ← Express API
│   ├─ index.js
│   ├─ prisma.js
│   ├─ middleware/
│   ├─ routes/
│   └─ utils/
├─ client/               ← React app (source)
│   ├─ index.html
│   └─ src/
├─ vite.config.js        ← builds client/ into ./public
├─ tailwind.config.js
└─ public/               ← generated by `npm run build` (gitignored)
```

---

## 1. Install

```bash
npm install
```

One install, one `node_modules`, for both server and client.

## 2. Create your Neon database

1. Sign up at [neon.tech](https://neon.tech) and create a new project.
2. Copy the **connection string** from the dashboard (Settings → Connection
   Details). Grab both the pooled and direct/unpooled string if Neon gives
   you both.

## 3. Configure environment variables

```bash
cp .env.example .env
```

Fill in `.env`:
```
DATABASE_URL="<neon pooled connection string>"
DIRECT_URL="<neon direct connection string>"   # paste the same value here if you only have one
JWT_SECRET="<any long random string, e.g. output of: openssl rand -base64 32>"
PORT=4000
```

## 4. Create the database tables

```bash
npm run prisma:migrate
```

## 5. Run it

**Local development** (hot reload for both server and client, one command):

```bash
npm run dev
```

This runs the API on `http://localhost:4000` and the React dev server on
`http://localhost:5173` (proxying `/api` to the backend) at the same time.
Open **http://localhost:5173**.

**Production-style, single server:**

```bash
npm run build   # builds client/ into ./public
npm start       # Express serves the API AND the built UI on one port
```

Open **http://localhost:4000** — everything is served from one process.

## 6. Use it

Create an account → add a course → add topics inside it → add subtopics
inside each topic → tick subtopics off. Hover any title to rename or delete
it.

## Deploying (single service, any Node host)

1. Push this repo to GitHub.
2. Create a **Web Service** on Render/Railway/Fly.io/etc., point it at the repo.
3. **Build command:** `npm install && npm run build`
4. **Start command:** `npm start`
5. **Environment variables:** `DATABASE_URL`, `DIRECT_URL`, `JWT_SECRET`
   (same as your `.env`).
6. Run the migration once against production, either from the host's shell
   or from your machine pointed at the prod `DATABASE_URL`:
   ```bash
   npm run prisma:deploy
   ```
7. Open the URL the host gives you.

## Deploying to Vercel

This repo is already set up for a **single Vercel project** — the React
build is served as static files, and the Express API runs as one
serverless function (`api/index.js` wraps `server/index.js` unchanged, and
`vercel.json` routes every `/api/*` request to it). No separate backend
host needed.

1. **Push this repo to GitHub** (or GitLab/Bitbucket).

2. **Import it on [vercel.com](https://vercel.com)** → New Project → select
   the repo. Leave the framework preset as **"Other"** — `vercel.json`
   already tells Vercel how to build it, so you don't need to change any
   build/output settings.

3. **Add environment variables** (Project Settings → Environment Variables,
   or in the import screen) — same values as your local `.env`:
   | Key | Value |
   |---|---|
   | `DATABASE_URL` | Neon **pooled** connection string |
   | `DIRECT_URL` | Neon **direct** connection string |
   | `JWT_SECRET` | any long random string |

   You don't need to set `PORT` or `CORS_ORIGIN` — Vercel assigns the port
   itself, and since the frontend and API are served from the same domain,
   the browser never makes a cross-origin request in the first place.

4. **Deploy.** Vercel runs `npm install` (which triggers the `postinstall`
   script → `prisma generate`, so the Prisma Client is always in sync with
   your schema), then `npm run build` (→ `vite build` → `public/`), then
   serves `public/` as static files with `api/index.js` handling `/api/*`.

5. **Run the migration against your Neon database once**, from your own
   machine, pointed at the *production* `DATABASE_URL`/`DIRECT_URL`:
   ```bash
   npm run prisma:deploy
   ```
   (Do this any time you change `prisma/schema.prisma` and redeploy, too.)

6. Open the `*.vercel.app` URL Vercel gives you — done.

**Redeploys:** every push to your main branch redeploys automatically;
Prisma Client regenerates every time via `postinstall`, so schema changes
never get "stuck" on an old client — just remember step 5 if the schema
itself changed.



| Method | Path | Description |
|---|---|---|
| POST | `/api/auth/register` | Create account, returns JWT |
| POST | `/api/auth/login` | Log in, returns JWT |
| GET | `/api/courses` | Full nested tree for the logged-in user |
| POST | `/api/courses` | Create a course |
| PUT | `/api/courses/:id` | Rename / edit description |
| DELETE | `/api/courses/:id` | Delete a course (cascades) |
| POST | `/api/courses/:courseId/topics` | Add a topic |
| PUT | `/api/topics/:id` | Rename a topic |
| DELETE | `/api/topics/:id` | Delete a topic (cascades) |
| POST | `/api/topics/:topicId/subtopics` | Add a subtopic |
| PUT | `/api/subtopics/:id` | Rename a subtopic |
| PATCH | `/api/subtopics/:id/toggle` | Tick / untick — triggers the auto-complete cascade |
| DELETE | `/api/subtopics/:id` | Delete a subtopic |

All routes except `/auth/*` require `Authorization: Bearer <token>` and are
scoped to the requesting user (ownership is checked on every write, so one
user can never edit another's data).
